# 漫画阅读器开发文档

> 文档性质：持续迭代的开发文档。后续所有产品决策、技术取舍、数据结构变更、阶段任务都优先记录在本文档中。

## 1. 项目定位

本项目是一个专门用于 Windows 的本地漫画书库管理与阅读软件。

核心方向：

- 离线优先：不依赖联网，不做在线漫画源、不做云同步、不抓取远程元数据。
- 文件夹优先：不支持 zip / cbz / rar / cbr 等压缩包，漫画以本地文件夹内的图片序列为基本单位。
- 管理优先：重点是作者、标签、封面、阅读进度、筛选与本地书库整理。
- 阅读舒适：阅读器应支持键盘操作、沉浸模式、单页/双页/连续滚动等常用模式。
- 视觉气质：整体 UI 参考苹果图书，强调干净、卡片式、柔和、少干扰。
- 双语言显示：软件界面至少支持中文与英文显示，可在设置中切换。

## 2. 已确认需求

### 2.1 硬约束

- 目标平台：Windows。
- 最终交付形态：Windows `.exe`，优先支持安装包，后续可追加绿色便携版。
- 项目目录：`C:\Users\Admin\Desktop\MySelf\Manga`。
- 不支持 zip / cbz / rar / cbr。
- 不需要联网功能。
- 支持双语言显示。
- 支持软件内快捷键设置。
- 支持作者分类与筛选。
- 支持自定义标签。
- 支持把第一页或指定页设置为封面。

### 2.2 推荐 MVP

第一版只做最核心闭环：

- 扫描一个本地书库根目录。
- 每个漫画文件夹识别为一本书。
- 读取文件夹内图片并按文件名自然排序。
- 默认使用第一页作为封面。
- 支持手动指定封面页。
- 支持编辑标题、作者、标签。
- 支持按作者和标签筛选。
- 支持基础阅读器：上一页、下一页、跳页、单页、双页、适应宽度、适应高度。
- 支持阅读进度记忆。
- 支持中文 / 英文界面切换。
- 支持快捷键设置与保存。

## 4. 产品架构

### 4.1 推荐技术路线

当前采用：

- WPF / .NET 8 原生 Windows 桌面软件。
- 使用 C# 处理文件扫描、图片读取、缩略图缓存、SQLite 数据库。
- 使用 WPF 构建书架、阅读器、设置、快捷键、多语言等界面。
- UI 方向从“苹果图书式纸质书隐喻”调整为“本地漫画库 + 极简图片查看器”：书库负责管理，阅读器负责干净看图。
- 已读、未读、在读、搁置属于书库筛选维度，不作为侧栏一级页面；侧栏只保留主页、书库、收藏等高频入口。
- 漫画源文件只读访问，所有用户劳动数据必须写入 SQLite，并通过自动备份保护。

理由：

- 最终形态是真正的 Windows `.exe` 软件，不依赖浏览器预览。
- 更适合本地图片查看器、文件夹扫描和原地存档。
- WPF 对图片显示、滚动、缩放、快捷键和窗口管理支持直接。
- 测试版可以发布为可直接运行的 `.exe`。
- SQLite 文件适合本地离线元数据保存，便于原地备份、迁移和恢复用户 Tag。
- 阅读器不再追求纸质书视觉，避免压缩图片显示空间，控件应保持可隐藏、轻量、统一。

备选：

- Tauri + React：UI 迭代快，但底层仍是 WebView，不符合当前“必须是软件”的方向。
- Electron + React：开发更快，但体积和内存占用偏大。
- WinUI 3：Windows 原生感更强，但工程复杂度高于当前 MVP 需要。
- Qt：适合传统桌面软件，但不符合“苹果图书式现代 UI”的快速迭代诉求。

### 4.2 发布与 exe 交付

目标交付结果：

- 开发期：可通过本地命令启动调试版桌面程序。
- 测试期：生成可直接运行的 Windows `.exe`，不依赖安装流程，双击即可验证。
- 发布期：生成 Windows 安装包，用户双击安装后使用。
- 追加目标：生成绿色便携版 `.exe`，可放在任意目录直接运行。

WPF 路线下的交付策略：

- 使用 `dotnet publish` 生成 Windows `.exe`。
- 测试版应用数据默认写入 `.exe` 所在目录下的存档目录，便于直接运行和整体搬移。
- 漫画源文件只读访问，软件只保存元数据、封面缓存和阅读进度。
- 后续需要图标、应用名、版本号、安装目录、卸载信息等发布配置。

注意：

- 测试版采用原地存档策略，启动时在 `.exe` 同级目录创建数据目录。
- 安装版更适合正式使用，便携版更适合临时测试和移动硬盘场景。

### 4.3 模块划分

- `LibraryScanner`：扫描书库根目录，识别漫画文件夹。
- `BatchImportAnalyzer`：分析拖入的作者文件夹，递归查找可导入的漫画目录，并在首次导入时给出基础 Tag 预判。
- `LibraryDatabase`：管理书籍元数据、作者、标签、阅读进度、状态、收藏和数据库备份。
- `ImageProvider`：读取本地图片，提供页面路径与尺寸信息。
- `CoverService`：生成与缓存封面缩略图。
- `ImageLoader`：只加载当前阅读页，并限制解码尺寸，避免大图一次性占用过多内存。
- `NaturalPathComparer`：按自然排序处理 `1, 2, 10` 这类页码。
- `ReaderState`：保存当前阅读模式、页码、缩放、阅读方向。
- `ShortcutManager`：管理快捷键配置、冲突检测与持久化。
- `I18nManager`：管理中文 / 英文语言包与运行时切换。
- `SettingsService`：保存软件设置。

### 4.4 数据安全约束

- 用户手动录入的 `Tag / 作者 / 角色 / 简介 / 封面页 / 阅读次数 / 阅读状态 / 收藏` 属于高价值数据，优先级高于 UI 效果。
- 启动迁移、删除库记录、路径重定位、保存核心元数据前必须保留数据库备份。
- 自动备份保存在 `.exe` 同级 `MangaReader_Data/backups/`，保留最近 40 份。
- 删除库记录只删除软件数据库记录，绝不删除硬盘漫画源文件。
- 扫描、同步、再次导入、批量导入都不得覆盖已有用户手工 Tag；智能推断只能在新书首次创建时写入。
- 后续如增加批量编辑，必须先做备份或可撤销机制。

## 5. 本地文件组织规则

### 5.1 书库根目录

用户选择一个或多个本地目录作为漫画库根目录。

示例：

```text
MangaLibrary/
  作者A/
    作品01/
      001.jpg
      002.jpg
      003.png
    作品02/
      001.webp
      002.webp
  作者B/
    作品03/
      page_001.jpg
      page_002.jpg
```

### 5.2 识别策略

- 包含图片文件的目录识别为一本书。
- 目录名默认作为书名。
- 上级目录可作为作者候选值，但不强制。
- 图片按自然排序排列，例如 `1.jpg`、`2.jpg`、`10.jpg` 应按 1、2、10 排序。
- 普通同步会递归扫描书库根目录，识别所有直接包含图片的目录。
- 作者批量导入会递归分析拖入的作者文件夹，找到所有直接包含图片的子目录作为候选漫画。

### 5.3 作者文件夹批量导入规则

- 用户可拖入一个或多个作者文件夹；拖拽数据中的所有文件夹都必须进入处理队列，不能只取第一个。
- 拖入文件夹时必须有明确视觉反馈，包括覆盖层、状态栏提示和复制光标。
- 每个作者文件夹会弹出批量导入确认窗口，允许修改作者名与每本漫画标题。
- 弹窗统计必须显示实际识别到的候选漫画数量，而不是只统计第一层子文件夹。
- 导入写入以 `folder_path` 为唯一合并依据；同一路径再次导入是更新，不同路径必须保留为不同漫画。
- 导入过程中不能清空当前书库集合，否则连续拖入多个作者文件夹会造成“覆盖”或“只剩一本”的体验。
- 已有书籍再次导入时保留原 Tag；智能推断结果只对新书首次创建生效。

后续可增强：

- 支持手动合并系列。
- 支持从目录名解析作者、系列、卷数。

## 7. 数据保存策略

### 7.1 总原则

- 漫画源文件默认只读访问，不在漫画图片目录内写入元数据。
- 测试版启动时在 `.exe` 所在目录生成存档目录，用户在软件内产生的数据统一保存到该目录。
- 结构化数据使用 SQLite 保存，便于查询、筛选、备份和迁移。
- 可再生成的数据使用缓存目录保存，例如封面缩略图。
- 关键用户配置使用设置文件或 SQLite 保存，但必须有统一入口管理。
- 所有写入操作应通过 Repository / Service 层完成，不允许 UI 组件直接写文件或数据库。

### 7.2 测试版数据目录

测试版 `.exe` 目标是直接运行，因此采用原地存档策略：程序启动时检查 `.exe` 所在目录，并在同级创建一个固定的数据目录。

建议策略：

- 测试版优先使用 `.exe` 同级的 `MangaReader_Data/` 目录。
- 如果目录不存在，启动时自动创建。
- 如果目录已存在，直接读取其中的数据库、设置和缓存。
- 整个测试版文件夹可以整体复制到另一台电脑，数据跟随移动。
- 如果 `.exe` 所在目录无写入权限，应弹出错误提示并引导用户移动到可写目录。
- 首次启动时自动创建数据目录、数据库文件和缓存目录。
- 数据目录路径需要在设置页中展示，方便用户备份和排查。

建议目录结构：

```text
MangaReader.exe
MangaReader_Data/
  app.db
  settings.json
  cache/
    covers/
  logs/
```

### 7.3 SQLite 保存内容

SQLite 保存所有需要稳定持久化和查询的数据：

- 书籍信息：标题、作者、路径、页数、封面页、收藏状态。
- 作者信息：作者名、别名、备注。
- 标签信息：标签名、颜色。
- 书籍与标签关系。
- 阅读进度：最后阅读页、阅读状态、最后打开时间。
- 快捷键绑定：功能 ID、快捷键组合、作用域、启用状态。
- 书库根目录：用户添加过的本地漫画库路径。

### 7.4 设置文件保存内容

`settings.json` 保存启动时需要快速读取、但不适合频繁查询的配置：

- 当前语言：`zh-CN` 或 `en-US`。
- 当前主题：浅色、深色、跟随系统。
- 窗口大小与位置。
- 最近打开的页面。
- 当前数据目录路径。

如果某项设置需要被筛选或复杂查询，则应迁移进 SQLite。

### 7.5 缓存保存内容

缓存目录只保存可重建内容：

- 封面缩略图。
- 页面尺寸索引。
- 后续可能的低清预览图。

缓存原则：

- 删除缓存不应导致用户数据丢失。
- 封面缓存文件名应基于 `book_id`、`cover_page_index` 和源文件修改时间生成。
- 当用户修改封面页时，应主动刷新对应缓存。
- 切换漫画时优先读取已有封面缓存；缓存不存在或源图变更时再生成。
- 阅读页本身不应全部缓存为位图，阅读器只加载当前页，必要时预留“上一页/下一页预读”。

### 7.6 保存时机

不同操作使用不同保存策略：

- 编辑作者、标题、标签：用户确认后立即写入 SQLite。
- 设置封面页：确认后立即写入 SQLite，并刷新封面缓存。
- 阅读进度：翻页时可防抖保存，例如 1-3 秒内合并写入；退出阅读器时强制保存一次。
- 快捷键修改：确认无冲突后立即保存。
- 语言切换：立即保存并刷新界面。

### 7.7 数据安全与恢复

- SQLite 写入应使用事务，避免批量操作中途失败导致数据半写入。
- 重新扫描书库不应删除用户手动编辑过的标题、作者、标签和封面设置。
- 批量导入不得把当前内存书库清空后只显示本次导入结果；应按 `FolderPath` 合并更新，保证连续导入不会让之前导入的漫画从列表中消失。
- `books.tags` 是每本漫画 Tag 的真实保存位置；`managed_tags` 只是候选池和管理视图。
- 默认预设标签被删除或重命名后，需要写入 `suppressed_tags`，避免重启后硬编码预设反弹。
- 如果漫画文件夹移动，应标记为“路径失效”，不要直接删除书籍记录。
- 提供“重新定位文件夹”功能用于修复移动后的路径。
- 后续可提供导出/导入元数据功能，方便迁移到另一台电脑。

## 8. 双语言显示方案

### 8.1 基础原则

- 所有 UI 文案不得硬编码在组件里。
- 使用语言 key，例如 `library.allBooks`、`reader.nextPage`。
- 中文为主语言，英文为第二语言。
- 用户可在设置中切换语言。
- 切换语言后界面即时刷新，不要求重启。

### 8.2 语言文件示例

```text
locales/
  zh-CN.json
  en-US.json
```

示例 key：

```json
{
  "library.allBooks": "全部书籍",
  "library.authors": "作者",
  "library.tags": "标签",
  "reader.nextPage": "下一页",
  "reader.previousPage": "上一页",
  "settings.shortcuts": "快捷键"
}
```

```json
{
  "library.allBooks": "All Books",
  "library.authors": "Authors",
  "library.tags": "Tags",
  "reader.nextPage": "Next Page",
  "reader.previousPage": "Previous Page",
  "settings.shortcuts": "Shortcuts"
}
```

### 8.3 需要注意的问题

- 英文文本长度通常更长，按钮和侧边栏需要预留弹性布局。
- 标签名、作者名、书名是用户数据，不参与界面翻译。
- 快捷键动作名需要翻译，但快捷键组合本身不翻译。

## 9. UI 方向

### 9.1 书架页

- 左侧边栏：全部书籍、最近阅读、收藏、作者、标签、设置。
- 顶部区域：搜索框、排序、筛选、视图密度。
- 主区域：大封面卡片网格。
- 卡片内容：封面、标题、作者、阅读进度。
- 视觉关键词：圆角、柔和阴影、大留白、低饱和背景。

### 9.2 详情页 / 侧面板

- 显示封面、标题、作者、标签、页数、阅读状态。
- 支持设置封面页。
- 支持编辑作者。
- 支持添加 / 删除标签。

### 9.3 阅读器

- 默认沉浸式。
- 鼠标移动时显示顶部或底部工具栏。
- 支持单页、双页、连续滚动。
- 支持从左到右 / 从右到左。
- 支持快捷键翻页。
- 自动保存阅读进度。

### 9.4 设置页

- 常规设置：语言、主题、默认书库路径。
- 阅读设置：默认阅读模式、默认方向、背景色。

### 12.3 风险点

- 大型图片书库扫描可能卡顿，需要后台任务与进度显示。
- 大量封面生成可能占用 CPU，需要队列化与缓存。
- 图片命名不规范会影响排序，需要可手动修正或重新排序。
- 阅读进度频繁写入可能影响性能，需要防抖保存与退出时强制保存。
- 重新扫描书库可能覆盖手动元数据，需要明确“扫描结果”和“用户编辑数据”的优先级。
- 批量导入如果只取第一个拖拽路径或导入前清空当前集合，会造成连续导入互相覆盖，必须按路径合并。
- 拖拽导入如果没有进入态反馈，用户会误以为操作失败，书库页必须保留可见覆盖层提示。
- 双页模式需要处理横图、跨页图、奇偶页偏移。
- 中英文 UI 长度差异会影响布局，需要早期就按 i18n 思路设计。
- 文件夹移动后不能删除数据库记录，应标记路径失效，并提供重新定位入口。

## 15. 架构审查与优化建议

> 基于 0.3.45 代码实际走读，记录当前架构问题与后续优化方向。

### 15.1 最大问题：MainWindow 是一个 God Class

`MainWindow.xaml.cs` 目前约 2,100 行，承担了以下所有职责：

- 书库加载与扫描
- 批量导入与智能 Tag 推断
- Tag 的增删改查、重命名、批量更新
- 筛选（作者、状态、Tag、搜索）与排序
- 主页书架数据聚合
- 11 个 ObservableCollection 的状态管理
- 导航视图切换（主页 / 书库 / 标签）
- 阅读窗口的启动

**影响：** 任何一个功能的小改动都可能触碰到毫不相关的其他逻辑。目前不崩溃靠的是迭代足够快，一旦要加语言切换、主题、设置页，这个文件会变得无法维护。

**方向：** 逐步提取 ViewModel 和 Service 层。WPF 不要求 MVVM 完全纯粹，但至少应该把 **数据操作** 和 **UI 事件响应** 拆开。

---

### 15.2 Tag 系统存在 O(N²) 性能瓶颈

`CreateTagChip()` 为每一个 Tag 芯片都调用了 `GetTagUsageCount()` 和 `GetTagBooks()`，这两个方法都是在内存的 `Books` 集合里遍历全部书籍来统计的。

`RefreshVisibleTags()` 和 `RefreshTagManagementItems()` 每次刷新都会重复触发这个逻辑。在筛选、导入、Tag 变更后这几个方法会被连续调用多次，等同于：

```
调用次数 = Tag 数量 × 全部书籍数 × 每次操作触发的刷新次数
```

书库规模达到几千本 + 几十个 Tag 之后，筛选操作会出现明显卡顿。

**修法：** 只在书库数据发生实质变化时重建一次 `tag → book[]` 的映射字典，之后 `GetTagUsageCount` 直接查字典，从 O(N) 降为 O(1)。

---

### 15.3 每次操作后连续执行五次全量刷新

书库筛选或数据变更后，当前代码会依次调用：

```
RefreshVisibleTags()
RefreshTagManagementItems()
RefreshAuthorFilters()
ApplyBookSort()
RefreshBookFilter()
```

这五步都是对 `Books` 集合的独立遍历，而且它们经常被同一个用户动作连续触发。等于对几千本书做了五遍全量扫描。

**修法：** 把这些刷新合并为一个统一的 `InvalidateView()` 入口，并在同一帧内只执行一次，用脏标记或合批的方式合并多次触发。

---

### 15.4 SQLite 连接每次使用都新建

`LibraryDatabase` 目前的 `Open()` 方法每次操作都新建一个 `SqliteConnection`。虽然 SQLite 有内置连接池，但每次的连接打开/关闭开销在批量导入时仍然可观。更重要的是，**没有开启 WAL 模式**，在写入中途崩溃可能导致数据库损坏。

**修法：** 启用 WAL 模式（`PRAGMA journal_mode=WAL`）；考虑在 Service 层持有单一长连接，只在批量操作结束后提交事务。

---

### 15.5 缺少关键数据库索引

`books` 表目前没有索引，全部筛选都靠把所有书读出来后在 C# 层过滤。以下列是高频筛选字段，应该加索引：

- `author`
- `reading_status`
- `is_favorite`
- `is_hidden`

**修法：** 在数据库迁移时用 `CREATE INDEX IF NOT EXISTS` 创建，数据量大之后筛选性能会有数量级的提升。

---

### 15.6 封面缓存没有淘汰机制，内存会无限增长

`CoverCache.LoadOrCreate()` 返回 `BitmapImage` 之后没有释放机制。加载几千张封面缩略图后内存占用会持续增长，从不回收。

**修法：** 用 `WeakReference<BitmapImage>` 做二级缓存，或者只在当前可视区域附近的封面保持强引用，滚出视口的及时释放。

---

### 15.7 Tag 颜色逻辑在两处各自维护

`MainWindow.xaml.cs` 的 `TagPresets` 硬编码了分类色，`MangaBook.cs` 的 `TagColor()` 方法也有一套独立的颜色映射。两套逻辑分散，后续修改颜色时容易漏改一处。

**修法：** 把 Tag 分类与颜色的定义抽为一个静态类或 JSON 配置，两处都从同一个来源读取。

---

### 15.8 批量导入的错误处理不健全

作者批量导入中，如果某一本漫画文件夹在处理过程中发生异常（文件锁定、权限不足等），整个导入任务会失败，没有"跳过这本、继续导入其余"的能力，也没有失败报告。

**修法：** 逐本捕获异常，导入结束后在弹窗里汇报"成功 N 本 / 失败 M 本"，允许用户查看失败原因。

---

### 15.9 服务之间没有接口隔离，无法测试

当前 `LibraryDatabase`、`LibraryScanner`、`CoverCache` 都是直接 `new()` 出来并通过字段持有，没有抽取接口。这意味着任何涉及文件系统或数据库的逻辑都无法在不启动真实环境的情况下验证。

**修法：** 不需要立刻上 DI 框架，但至少给核心 Service 各提取一个 `IXxx` 接口，便于后续写集成测试时可以替换 Fake 实现。

---

### 15.10 优先级建议

上面的问题按影响程度排序：

| 优先级 | 问题 | 为什么先做 |
|--------|------|----------|
| P0 | 开启 SQLite WAL 模式 | 防止崩溃损坏数据，一行代码 |
| P0 | 给高频字段加数据库索引 | 书库变大后筛选直接卡死 |
| P1 | 合并多次全量刷新 | 用户能感知到的卡顿，改动局限 |
| P1 | 修复 Tag 统计的 O(N²) | 同上，规模一大就暴露 |
| P1 | 批量导入错误处理 | 当前导入失败无任何反馈 |
| P2 | 封面缓存淘汰 | 长时间使用后内存问题 |
| P2 | 统一 Tag 颜色定义 | 维护成本，当前已在两处 |
| P3 | 提取 ViewModel / Service | 长期可维护性，当前功能已够用 |

---

### 16.1 搜索框没有防抖，每次击键触发全量过滤

`BookSearchBox_TextChanged`、`TagSearchBox_TextChanged` 这两个事件没有任何延迟，直接调用全量过滤和标签刷新。用中文输入法输入一个词需要打多个字母，每个字母都会触发一次对整个书库的遍历。

**修法：** 用 `DispatcherTimer` 做 200-300ms 防抖，停止输入后再触发一次刷新，击键过程中不刷新。

---

### 16.2 SaveBookTagsBatch 是 N+1 循环写入

`LibraryDatabase.SaveBookTagsBatch()` 对每本书执行一条独立的 UPDATE SQL，虽然有事务包裹，但每次进入循环都要通过连接发送一条语句。批量操作（如全局重命名 Tag）涉及书籍数量大时，这个开销是线性的。

**修法：** 用一条带 CASE 的 SQL 或 VALUES 临时表一次性批量更新，把 N 次 SQL 合并为 1 次。这个优化在书库几百本以上时效果明显。

---

### 16.3 MangaBook.NotifyAll() 一次触发 30+ 个属性通知

`MangaBook.cs` 的 `NotifyAll()` 方法列出了 30+ 个 `OnPropertyChanged()` 调用，每次书籍数据有任何变化都会触发这些通知，WPF 的 UI 绑定会响应每一个，导致同一帧内多次重绘。

**修法：** 用 `null` 参数的 `OnPropertyChanged(null)` 触发一次全刷，而不是手动列举所有字段名。如果需要更精确，只通知真正发生变化的属性。

---

### 16.4 魔法数字/字符串散落全文，应替换为枚举

以下地方用裸字符串或索引数字编码了状态，后续加新值容易漏改：

- 导航视图：`”home”` / `”library”` / `”tags”` 字符串（`_currentNavigationKey`）
- 排序方式：下拉框索引 0-5，`ApplyBookSort()` 里用 switch case 数字
- 阅读状态：`”reading”` / `”finished”` / `”unread”` / `”paused”` 字符串字面量，在 MainWindow 和 MangaBook 两处各写了一遍
- 书本样式：索引 0-3，`BookStyleIndex = BookStyle % 4`，含义完全依赖注释

**修法：** 定义 `NavigationView`、`SortMode`、`ReadingStatus`、`BookStyleKind` 四个枚举，改完之后编译器会帮你检测遗漏的 case。

---

### 16.5 批量导入错误处理缺失

`ImportAuthorBatchAsync()` 在导入每本书时，如果文件夹有权限问题、文件被锁定或图片无法读取，单本失败不会被跳过，也不会在结束后汇报哪几本失败了。

**修法：** 逐本 try-catch，失败的记录错误信息继续处理下一本，结束后在弹窗显示”成功 N 本 / 失败 M 本”和原因列表。

---

### 16.6 XAML 前端问题

#### 颜色值散落全文，无中央色板

`MainWindow.xaml` 全文出现了几十处十六进制颜色值（`#111827`、`#F6F7F9`、`#E5E7EB`、`#374151` 等），每处直接内联在控件属性里。现在改一个背景色需要全局搜索并逐个修改，很容易漏。

**修法：** 在 `App.xaml` 的 `ResourceDictionary` 里定义命名 `SolidColorBrush`（如 `AppBackground`、`BorderLight`、`TextPrimary`），控件改用 `{StaticResource xxx}` 引用。

#### Empty-state 面板重复 4 次

主页、书库、标签页、各分区都有自己的空状态提示 UI，结构几乎一样（居中文字 + 可选按钮），但写了 4 套独立的 XAML，改一处样式要同步改 4 处。

**修法：** 抽一个 `EmptyStatePanel` UserControl，接受 Message、ActionLabel、ActionCommand 几个属性，复用到所有空状态场景。

#### ComboBox 自定义模板有两套，逻辑几乎相同

`FilterPillComboBox` 和普通 ComboBox 的自定义模板代码结构相同，只是外观微调，但写了两份完整模板，任何行为改动需要同步两份。

**修法：** 合并为一个模板，用 `TemplateBinding` 或附加属性区分外观差异。

#### 书本卡片 4 种样式用 DataTrigger 嵌套，XAML 体积庞大

4 种样式（Groove、Bound、Plain、WhiteBorder）通过 `DataTrigger` 在同一个 `ControlTemplate` 内嵌套切换，这一块撑起了大量 XAML 行数，阅读困难。

**修法：** 提取为 4 个独立 DataTemplate 或 UserControl，用 `DataTemplateSelector` 根据 `BookStyleIndex` 选择，每个样式单独维护。

## 14. 当前结论

建议把本项目做成一个”干净、离线、文件夹驱动”的漫画书库软件，而不是综合阅读器。

最关键的产品边界：

- 不联网。
- 不支持压缩包。
- 不做云同步。
- 不做复杂在线元数据。
- 只围绕本地漫画文件夹、作者、标签、封面、阅读体验打磨。

技术上当前采用 WPF / .NET 8 + SQLite，以获得真正的 Windows 原生 `.exe` 体验。
